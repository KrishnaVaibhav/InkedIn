# InkedIn Research and Architecture

Last updated: 2026-07-04

## Purpose

This document defines a buildable architecture for a local manga/comic colorization app that:

- Runs first-class on Windows PCs without WSL.
- Also runs on Linux and macOS.
- Uses GPU acceleration where the host supports it.
- Keeps user images, books, models, logs, and paths local by default.
- Treats PDFs, images, archives, model files, and UI input as hostile until validated.
- Controls memory, GPU memory, temp files, and worker lifetime to prevent practical leaks.

Important constraint: no software architecture can honestly promise "zero leaks" or "zero loopholes". The correct target is a small trusted core, explicit threat model, strict boundaries, automated security tests, and process isolation so failures are contained.

## Product Scope

InkedIn should colorize manga/comic input locally and export the result without requiring cloud processing.

Supported input targets:

- Single images: `png`, `jpg`, `jpeg`, `webp`, `bmp`, `tiff`.
- Books: `pdf`, `cbz`.
- Optional later: `cbr`, because RAR handling usually needs extra native binaries and creates a larger attack surface.

Supported output targets:

- Image folder.
- `pdf`.
- `cbz`.
- Same-format export when safe.

Core user flows:

- Import one image, a folder, PDF, or CBZ.
- Preview pages and select specific pages.
- Choose mode: fast automatic, reference-based, or themed/prompted.
- Run locally on CPU or GPU.
- Cancel, resume, retry failed pages, and export.

## Top-Level Decision

Use a desktop shell with a hardened local ML worker:

- UI shell: Tauri v2 + TypeScript frontend.
- Native shell/orchestrator: Rust via Tauri commands.
- ML/image engine: Python package run as a supervised local subprocess.
- IPC: stdio JSON-RPC or a named pipe/unix socket, not an open HTTP port.
- Job state: local SQLite database.
- Models: local model cache with signed/hashed manifests.

Why this split:

- Tauri provides a small cross-platform desktop shell and a permission/capability model.
- Rust is a good trust boundary for filesystem access, process supervision, path normalization, and safe IPC.
- Python remains the practical ecosystem for PyTorch, diffusion models, image processing, and research model integration.
- Avoiding a local web server removes a common leak path. If a FastAPI server is used during development, it must bind only to `127.0.0.1`, use a random per-session auth token, and be disabled in packaged production builds.

## Pipeline

Every page should move through the same bounded pipeline:

```text
Select input
  -> validate file type and size
  -> create per-job temp workspace
  -> extract pages without trusting archive paths
  -> decode page into normalized image
  -> detect panels
  -> detect text/bubbles/SFX masks
  -> handle screentone/shading
  -> colorize selected panels/pages
  -> enforce cross-page character/style consistency
  -> composite text and line art back
  -> optional upscale
  -> export
  -> clean temp workspace
```

Implementation rules:

- Stream pages. Do not load a full book into RAM.
- Use per-page intermediate files with deterministic names generated by the app, never names from archives.
- Store temporary data under a per-job directory inside the app temp root.
- Delete temp data on successful export, cancellation, and startup cleanup of stale jobs.
- Keep original input immutable.
- Export to a new file path first, then atomically move/rename when complete.

## Model Strategy

Ship three colorization modes behind one interface.

| Mode | Purpose | Baseline implementation | GPU expectation |
| --- | --- | --- | --- |
| Fast automatic | Bulk colorization, low VRAM | Lightweight GAN or manga-colorization-v2 style model | CPU usable, GPU faster |
| Reference-based | Consistent characters/style from reference pages | MangaNinja-style/reference diffusion model | GPU strongly preferred |
| Theme/prompt | Presets like pastel, noir, warm sunset | Stable Diffusion/ControlNet-style line-art pipeline | GPU required for acceptable speed |

Model wrapper contract:

```text
load(model_id, device, precision)
warmup(sample_shape)
infer(page_or_panel, controls, cancellation_token)
unload()
health()
```

Rules:

- Prefer `safetensors` or ONNX model files.
- Do not load arbitrary pickle checkpoints from untrusted sources.
- If legacy PyTorch checkpoints are unavoidable, load only from trusted manifests and use `torch.load(..., weights_only=True)` where compatible.
- Every model artifact must have a manifest containing `model_id`, source URL, license, expected SHA-256, file size, supported devices, supported precision, and minimum app version.
- User-supplied models are disabled in MVP. If added later, they must run under the same manifest and sandbox rules.

## GPU Backend Matrix

There is no single GPU backend that covers Windows, Linux, and macOS equally. Use a device abstraction and pick the best supported provider at runtime.

| OS | GPU | Primary path | Secondary path | Notes |
| --- | --- | --- | --- | --- |
| Windows | NVIDIA | PyTorch CUDA | ONNX Runtime CUDA | Best target for diffusion and reference models. |
| Windows | AMD/Intel/Qualcomm | ONNX Runtime DirectML for ONNX-exported models | CPU | PyTorch CUDA does not apply. DirectML is broad, but operator coverage must be tested per model. |
| Linux | NVIDIA | PyTorch CUDA | ONNX Runtime CUDA | Best Linux path. |
| Linux | AMD | PyTorch ROCm | CPU or ONNX/OpenVINO for supported hardware | Treat ROCm support as hardware and distro specific. |
| Linux | Intel | OpenVINO/ONNX where applicable | CPU | Useful for GAN/upscale models if exported. |
| macOS | Apple Silicon/AMD | PyTorch MPS | ONNX Runtime CoreML or CPU | MPS is the PyTorch path; CoreML is useful only for exported models. |

Runtime device selection:

```text
1. Read user preference: Auto, CPU, CUDA, MPS, ROCm, DirectML, CoreML.
2. Probe available backends.
3. Check model manifest support.
4. Run a tiny warmup inference.
5. If warmup fails, fall back to the next safe backend and record why.
6. Never silently switch from local GPU to cloud. Cloud processing is out of scope.
```

Precision policy:

- CUDA: prefer `fp16` for diffusion; allow `bf16` on supported newer GPUs; fall back to `fp32`.
- MPS: start with `fp16` only after model validation; keep `fp32` fallback because some ops/models behave differently.
- DirectML/CoreML/ONNX: precision depends on exported graph and provider support.
- CPU: use `fp32` unless a tested quantized path exists.

VRAM policy:

- Detect total and free VRAM when possible.
- Use tile/panel inference for large pages.
- Set batch size from a warmup profile, not a fixed constant.
- Recycle the inference worker after a configurable number of pages or when memory crosses a high watermark.
- Free model handles on mode switch and after fatal inference errors.

## Windows-First Build Plan

Windows must work without WSL, Docker, or admin rights for normal use.

Installer:

- Tauri NSIS/MSI installer.
- App installed under `Program Files` or user-local install path.
- App data under `%LOCALAPPDATA%\InkedIn`.
- Model cache under `%LOCALAPPDATA%\InkedIn\models`.
- Temp jobs under `%LOCALAPPDATA%\InkedIn\jobs`.

Runtime:

- Bundle the Tauri app.
- Bundle or bootstrap a private Python runtime.
- Create a private virtual environment for Python dependencies.
- Install CPU wheels by default.
- Offer a first-run GPU setup screen that installs the matching CUDA/DirectML profile after checking hardware.

GPU profiles:

- `cpu`: smallest, safest baseline.
- `cuda`: NVIDIA Windows machines.
- `directml`: ONNX-only acceleration on broad Windows GPU hardware.

Do not bundle all GPU stacks in one installer. It makes the installer huge, increases update risk, and causes dependency conflicts.

## Linux Build Plan

Linux targets:

- AppImage for broad testing.
- `.deb` for Ubuntu/Debian users.
- `.rpm` later if needed.

Runtime paths:

- App data: `~/.local/share/inkedin`.
- Config: `~/.config/inkedin`.
- Cache: `~/.cache/inkedin`.

GPU profiles:

- `cpu`: default.
- `cuda`: NVIDIA with matching driver.
- `rocm`: AMD on supported distros/GPUs.
- Optional `openvino`: Intel acceleration for exported ONNX models.

Linux caution:

- Do not assume ROCm works on every AMD GPU.
- Do not assume distro-provided Python packages match PyTorch wheel requirements.
- Keep Python runtime and dependencies private to the app where practical.

## macOS Build Plan

macOS targets:

- Apple Silicon build.
- Intel build only if there is demand.
- Universal build only if dependency size and signing stay manageable.

Runtime paths:

- App data: `~/Library/Application Support/InkedIn`.
- Cache: `~/Library/Caches/InkedIn`.

GPU profiles:

- `mps`: PyTorch MPS for Apple GPU acceleration.
- `coreml`: only for exported ONNX/CoreML-compatible models.
- `cpu`: fallback.

Packaging:

- Signed and notarized `.app`.
- No unsigned helper binaries.
- The Python worker must be included in signing/notarization rules.

## Security Architecture

### Trust Boundaries

Untrusted:

- Imported files.
- Archive member names and metadata.
- PDF structure.
- Image metadata.
- Downloaded model weights.
- Any text prompt or theme input.
- Frontend state.

Trusted core:

- Rust orchestrator.
- Signed app binaries.
- Checked-in model manifests.
- Python worker code shipped with the app.

### Filesystem Controls

- UI never receives broad filesystem access.
- User selects input/output through native dialogs.
- Rust canonicalizes and validates all paths before passing them to Python.
- Python receives job-scoped paths and cannot request arbitrary filesystem access through IPC.
- Archive extraction never writes using archive member paths.
- All output filenames are generated or sanitized by the Rust layer.
- Prevent overwriting input files unless the user explicitly confirms and the app writes through a safe temp file first.

### Archive Controls

For `cbz`:

- Accept only ZIP containers with image entries.
- Reject absolute paths.
- Reject `..` path traversal.
- Reject symlinks and special files.
- Enforce max entries.
- Enforce max compressed size.
- Enforce max expanded size.
- Enforce max expansion ratio to block zip bombs.
- Decode entries from memory/stream into generated temp paths.

For `cbr`:

- Defer to a later milestone unless needed.
- If added, use a sandboxed extractor process and the same expansion/path rules.
- Never shell out with user-controlled filenames.

### PDF and Image Controls

- Enforce max file size before decoding.
- Enforce max page count.
- Enforce max image dimensions and megapixels.
- Strip metadata by default on export.
- Normalize orientation and color profile.
- Render PDFs in a short-lived decode worker so parser crashes/leaks do not poison the inference worker.
- Keep decode libraries updated and covered by dependency scanning.

### Model Supply Chain Controls

- Model downloads use HTTPS.
- Every download must match SHA-256 in a signed or checked-in manifest.
- Store model files read-only after download where the OS permits.
- Do not execute code from model repositories.
- Prefer `safetensors` and ONNX.
- Avoid `trust_remote_code`.
- Avoid dynamic imports from model folders.
- Track licenses per model.

### IPC Controls

- Prefer stdio JSON-RPC between Rust and Python.
- Use schema-validated messages.
- Include job id, operation id, and cancellation token in every command.
- Reject unknown commands and unknown fields.
- Impose message size limits.
- Do not send raw image bytes over IPC unless necessary; pass job-scoped file handles/paths.
- If a local socket is required, bind to localhost only and use a random per-session token.

### Network Controls

Default behavior:

- No telemetry.
- No cloud processing.
- No remote logging.
- No open listening port.
- No network access from inference jobs.

Allowed network actions:

- Explicit model download.
- Explicit update check.
- Explicit bug report export chosen by the user.

All network actions must be visible in UI and cancellable.

### Logging Controls

- Logs are local only.
- Redact full user paths by default.
- Never log image pixel data, model tokens, environment variables, or full prompts unless debug mode is explicitly enabled.
- Debug bundles must show the user exactly what files are included before export.

## Memory and Resource Leak Controls

### Process Model

Use process isolation as the hard leak boundary:

- Rust orchestrator is long-lived.
- Decode workers are short-lived.
- Inference worker can be long-lived for model reuse, but must be recyclable.
- Export worker is short-lived.

Recycle inference worker when:

- A job finishes and user preference says "release GPU after job".
- N pages have run since worker start.
- GPU memory remains above threshold after cleanup.
- A model switch occurs.
- An unhandled exception occurs.

### Python Rules

- Use context managers for every file handle.
- Avoid global mutable state except the active model registry and device manager.
- Use `torch.inference_mode()` for inference.
- Explicitly close PIL/OpenCV/PyMuPDF resources.
- Delete large tensors after each page/panel.
- Synchronize GPU before measuring memory.
- Use process restart as the final GPU cleanup, not only `empty_cache()`.

### Data Rules

- Do not keep all page images in memory.
- Cache thumbnails separately from full-resolution pages.
- Store intermediate full-resolution images on disk under the job temp directory.
- Use bounded queues between pipeline stages.
- Backpressure the UI: do not allow unlimited jobs.

### Failure Rules

- Cancellation must interrupt future stages and let current native call finish safely.
- Failed pages should mark error state and continue if the user enabled partial export.
- Crash recovery scans stale job temp folders at startup and offers cleanup/resume.

## UI Security Rules

- Tauri capabilities must expose only required commands.
- No broad filesystem plugin permissions.
- Strict Content Security Policy.
- No remote scripts, remote fonts, or remote images in the app UI.
- Frontend validates for UX only; Rust/Python validate for security.
- Prompt/theme text is data only, never code.
- Disable developer tools in production builds.

## Job Database

Use SQLite locally.

Tables:

- `jobs`: id, created_at, status, input summary, output target, selected pages, mode, device.
- `pages`: job_id, page_number, status, source_hash, thumbnail_path, output_path, error_code.
- `models`: model_id, version, local_path, sha256, installed_at, last_verified_at.
- `events`: job_id, timestamp, level, redacted_message.

Rules:

- Store relative app-internal paths where possible.
- Store hashes for deduplication, not original image content.
- Keep schema migrations deterministic and tested.

## Quality Gates

No release should ship unless these pass.

Security tests:

- Path traversal in ZIP names.
- Absolute path entries in ZIP.
- Symlink entries in ZIP.
- Zip bomb expansion ratio.
- Oversized image dimensions.
- Fake extension with wrong magic bytes.
- Corrupt PDF.
- Corrupt image.
- Model hash mismatch.
- IPC unknown command.
- IPC oversized message.
- Output overwrite attempt.

Leak/resource tests:

- Colorize the same page 100 times and verify RSS does not grow unbounded.
- Run a 200-page book and verify temp files are cleaned.
- Cancel during decode, inference, and export.
- Kill the Python worker mid-job and verify the Rust shell recovers.
- Switch models repeatedly and verify GPU memory returns below threshold after worker recycle.

Cross-platform tests:

- Windows CPU smoke.
- Windows CUDA smoke on self-hosted NVIDIA runner.
- Windows DirectML smoke for ONNX model if supported.
- Linux CPU smoke.
- Linux CUDA smoke.
- Linux ROCm smoke only on supported self-hosted hardware.
- macOS CPU smoke.
- macOS MPS smoke on Apple Silicon.

Image quality tests:

- Golden input set with expected dimensions and structural similarity thresholds.
- Text mask preservation test.
- Panel boundary bleed test.
- Reference consistency test across repeated character pages.
- Export round-trip test for PDF and CBZ.

## Recommended Technical Choices

| Area | Choice | Reason |
| --- | --- | --- |
| Desktop shell | Tauri v2 | Cross-platform, smaller surface than Electron, capability controls. |
| UI | TypeScript + Vite + React or Svelte | Fast desktop UI iteration. Pick one and keep it simple. |
| Orchestrator | Rust | Strong path/process/IPC boundary. |
| ML runtime | Python + PyTorch | Best ecosystem for current colorization models. |
| Exported inference | ONNX Runtime | Useful for DirectML/CoreML/OpenVINO acceleration when models export cleanly. |
| Jobs | SQLite | Local, reliable, no broker. |
| IPC | stdio JSON-RPC | No open local port. Easy supervision. |
| Packaging | Tauri bundler + private Python runtime | Cross-platform app distribution. |
| Dependency locking | `uv.lock` for Python, Cargo.lock, package lockfile | Reproducible builds. |

Avoid in MVP:

- Celery/Redis. Too much local surface for a desktop app.
- Plugins. They create a large security policy problem.
- Cloud fallback. Violates privacy expectations.
- Arbitrary user model repositories. Too much supply-chain risk.
- CBR unless user demand justifies the native extractor risk.

## Design Review: Gaps Found and Locked Decisions (2026-07-03)

Review of this document found the following drawbacks. Each is now fixed with a locked decision.

### Gap 1: Models named but not locked

Fixed. Locked model plan for the three modes:

| Mode | Locked choice | VRAM (fp16) | Notes |
| --- | --- | --- | --- |
| Fast automatic | manga-colorization-v2 architecture (ResNeXt U-Net generator + feature extractor) | ~2-3 GB, CPU capable | Weights license is unclear (research repo). Ship behind manifest; plan retrain/distill for redistribution. |
| Reference-based | ~~MangaNinja (CVPR 2025)~~ **Superseded 2026-07-04: Cobra (SIGGRAPH 2025)** — see Research Update below | ~9 GB | Cobra: Apache-2.0, 640×1024, multi-reference, color hints. MangaNinja demoted to optional precision tool (CC BY-NC, 512px, single ref). Milestone 4. |
| Theme/prompt | SD 1.5 + ControlNet lineart (`lllyasviel/control_v11p_sd15_lineart`) | ~4-6 GB | SD 1.5 chosen over SDXL to fit 8 GB cards. Theme = prompt preset + optional LoRA. Milestone 4. |

### Gap 2: Panel detection method unspecified

Fixed. MVP uses rule-based OpenCV detection: adaptive threshold -> morphological close -> contour/gutter recursive cut, with fallback to whole-page when confidence is low. Do NOT use kumiko (AGPL-3.0 license contamination risk). A YOLO-seg panel model may replace this in Milestone 4 under the model manifest rules.

### Gap 3: Text/bubble mask method unspecified

Fixed. MVP protects text via line-luminance recomposite (Gap 7) which keeps black text crisp without detection. Milestone 4 adds a YOLO-based bubble/text detector (e.g. ogkalu/comic-text-and-bubble-detector class of models) so bubble interiors stay white and text is composited back losslessly.

### Gap 4: Screentone handling has no concrete method

Fixed. The fast-automatic model is trained on manga and consumes screentones natively; MVP passes tones through untouched. Keep a `pipeline/screentone.py` hook with a LoG (Laplacian-of-Gaussian) based tone-suppression filter available as a pre-process toggle for the diffusion path, where tones confuse SD lineart conditioning.

### Gap 5: Cross-page consistency has no mechanism

Fixed. Three-level plan:
- MVP: deterministic seed per job + identical model state across pages (no per-page randomness).
- MVP+: post-process palette anchoring — match per-page color histograms to a user-picked anchor page (LAB-space moment matching).
- Milestone 4: reference-based mode (MangaNinja) with a per-book reference cache; ColorFlow-style retrieval is the long-term direction.

### Gap 6: Diffusion resolution/tiling missing

Fixed. Diffusion mode runs panel-wise at max 1024 px long edge, upscales back with Real-ESRGAN or Lanczos, and blends seams with feathered masks. Full-page diffusion at native manga resolution (~1100x1600+) is out of spec for SD 1.5 and for 8 GB VRAM.

### Gap 7: Line-art preservation missing

Fixed. All modes end with line recomposite: convert model output to LAB, take chroma (A/B) from the model, take lightness (L) from a blend of model L and source page L weighted toward source in dark regions. Keeps ink lines and text sharp regardless of model behavior.

### Gap 8: Python bundling undecided

Fixed. Development and current builds use a workspace-local venv (`.venv/`) managed by `uv`. Packaged builds (Milestone 5) bundle python-build-standalone as a Tauri sidecar. No system/user site-packages are ever touched.

### Gap 9: Storage locality

Changed by owner requirement: ALL app storage lives inside the workspace, not under `%LOCALAPPDATA%`:

- venv: `D:\GitHub\InkedIn\.venv`
- uv cache: `D:\GitHub\InkedIn\.uv-cache` (`UV_CACHE_DIR`)
- model cache: `D:\GitHub\InkedIn\models\weights` (gitignored)
- jobs/temp: `D:\GitHub\InkedIn\data\jobs`
- job db: `D:\GitHub\InkedIn\data\inkedin.db`

The `%LOCALAPPDATA%` layout in the Windows Build Plan applies only to end-user installer builds, not this development workspace.

### Gap 10: MVP shell order

The Tauri + Rust + Python triple stack is the end-state. To get a working app sooner without violating the security rules, the MVP UI is the dev-mode local web UI this document already permits: FastAPI bound to `127.0.0.1` only, random per-session token required on every request, no external origins (strict CSP), disabled in packaged builds. The Rust/Tauri shell replaces it in Milestone 3+ without changing the Python core, because the pipeline is already IPC-shaped (job-scoped paths, schema-validated commands).

## Research Update (2026-07-04): Import Coverage and Comic-Accurate Colorization

Findings from a fresh survey of the mid-2026 landscape. Two questions: what should InkedIn ingest, and what produces the most comic-accurate (canon-faithful) color on local hardware.

### Colorization model landscape (mid-2026)

| Model | Venue | Base / arch | License | Cost measured | References | Code + weights | Verdict |

Key numbers that decided it: Cobra does what ColorFlow does at 1/4 the memory and 3x the speed, with a permissive license, purpose-built for comic pages, and it accepts both a reference library and per-point color hints — the two mechanisms comic-accuracy needs.

### Locked plan: comic-accurate coloring (supersedes Gap 1 reference row)

"Comic-accurate" means characters keep their canonical colors on every page. Mechanism stack, in pipeline order:

1. **Per-series reference library.** User imports colored canon material once per series: official color volume pages, covers, anime stills. Stored under `data/refs/<series>/`. 8-24 refs is the working set on 8-12 GB VRAM (Cobra scales linearly if the user has more headroom).
2. **Retrieval (ColorFlow's idea, our implementation).** Embed reference pages patch-wise with a CLIP image encoder (ViT-L, already shipped as part of the ai-mode stack); at job time embed each target page/panel and select top-k patches/pages by cosine similarity → feed as Cobra's reference set. Per-book index cached in the job DB.
3. **Character palette DB.** Optional JSON per series: `character → {hair, skin, outfit} hex`. Auto-placed as Cobra hint points on matched regions; doubles as a post-pass LAB palette snap for the GAN/SD backends. UI: eyedropper + character name, saved with the series.
4. **Existing invariants stay for every backend:** fixed seed per job, panel-to-panel palette anchor, chroma-only LAB recomposite with guided-filter edge snap (lines and text always survive the model).

Backend ladder by hardware:

| VRAM | Best mode |
| --- | --- |
| CPU only | `fast` GAN + palette snap |
| 8 GB | `ai` (SD1.5 + ControlNet + IP-Adapter, panel-wise) or Cobra with small ref set / CPU offload |
| 9-12 GB | **Cobra reference mode, 8-24 refs — the comic-accurate flagship** |
| 16 GB+ | Optional: Qwen-Image-Edit (quantized) instruction backend behind recomposite |

Near-term `ai`-mode upgrades (before Cobra lands): switch IP-Adapter to the Plus variant for stronger reference adherence, optional ControlNet-tile second pass for hi-res refinement, per-panel prompt overrides in the UI. Keep WD14 tagging (a 2026-03 SDXL paper independently validates tagger-driven conditioning).

### Panels, text, bubbles (Milestone 4 lock)

- **Adopt [ogkalu/comic-text-and-bubble-detector](https://huggingface.co/ogkalu/comic-text-and-bubble-detector)**: RT-DETR-v2 r50vd, 42.9M params, Apache-2.0, ONNX available (fits the onnxruntime dependency WD14 already uses). Use: bubble mask → keep interiors white; text masks → lossless text composite (closes Gap 3 exactly as planned).

Upgrade candidates: [Magi v2](https://huggingface.co/ragavsachdeva/magiv2) detects panels + characters + OCR + speaker attribution but is personal/research/non-commercial — optional plugin only, never bundled. CVPR 2025 published full segmentation annotations for Manga109 if we ever train our own panel-seg (Manga109 itself is research-only — check before use).
- Avoid ultralytics-YOLOv8-based bubble models (e.g. kitsumed/yolov8m_seg-speech-bubble) in the shipped app: AGPL lineage.

### Import coverage plan (priority order)

Current: images, folders, PDF, CBZ. Add:

1. **CB7** (7-Zip comics): `py7zr`, pure Python — no native extractor, lowest risk. Same limits as CBZ: max entries, compressed size, expanded size, expansion ratio; reject non-regular entries (py7zr's own symlink checks are best-effort, ours are authoritative). Read entries to memory, write via generated names only.
2. **CBT** (tar comics): stdlib `tarfile` with Python 3.12+ `filter="data"` (blocks absolute paths, traversal, symlinks, devices) — and still route every entry through our validators.
3. **CBR/RAR**: `rarfile` with backend preference **unar > bsdtar > unrar ≥ 6.12**. unrar < 6.12 has CVE-2022-30333 (path traversal via crafted Windows symlinks); bsdtar/libarchive lacks solid-archive and filter support so many CBRs fail on it. Mitigations: never let the tool extract to disk (stream entry bytes through `rarfile`'s read API), version-check the backend in `inkedin doctor`, run inside the short-lived decode worker. Windows ships no RAR tool by default — detect, degrade with a clear "install unar/unrar or convert to CBZ" message. This satisfies the original "sandboxed extractor + same expansion rules" requirement.
4. **EPUB** (huge share of digital manga): it is a ZIP — reuse all CBZ container rules, then parse `META-INF/container.xml` → OPF spine → ordered image resources. Import images only; ignore scripts, fonts, styles. Fixed-layout manga EPUBs map 1:1 to pages.
5. **Batch import**: folder-of-archives, recursion, multi-file drag-drop in UI; one job per book.
6. Import niceties: double-page spread split (aspect > 1 → optional gutter split), RTL page-order toggle, cover detection/skip.

Rejected: DjVu (native decoder surface, dying format — revisit on demand), MOBI/AZW3 (DRM entanglement — out of scope).

## Self-Reference Colorization Plan (2026-07-04) — Tier 0 IMPLEMENTED same day

Status: detection + skip + fast/theme palette anchoring + ai IP-Adapter auto-ref
shipped (nearest ref by L-moment distance; CLIP retrieval remains Tier 1).
Detection note: hue-spread metric must be circular std-dev in DEGREES with a
low threshold (8°) — a "fraction of resultant vector" metric rejects real
warm-dominant covers (measured 0.04 on a genuine multi-color page).
Also shipped as part of the same quality pass: chroma-void inpainting for
model-missed spots, ink-neutralization ramp lowered 90→65, chroma-only palette
matching option, and border-component filtering in detector bubble masks
(fixes rectangular white halos around round bubbles).

Books often ship with a few official color pages (volume openers, covers,
inserts). Detect them at ingest and use them as the color authority for the
rest of the book — the book colorizes itself.

### 1. Detection (cheap, deterministic, at ingest)

For each ingested page compute chroma stats in LAB on a downscaled copy
(the thumbnail pass already decodes every page):

- `colorful_frac` = fraction of pixels with chroma magnitude
  `sqrt((a-128)^2 + (b-128)^2) > 12`.
- `hue_spread` = circular std-dev of hue over those colorful pixels.

Classification:

| Result | Rule |
| --- | --- |
| **color page** | `colorful_frac > 0.10` and `hue_spread` high (multi-hue art) |
| **tinted scan** (yellowed paper, sepia print) | `colorful_frac` high but hue_spread ~0 (uniform cast) → treat as B&W |
| **B&W** | otherwise. JPEG chroma noise stays far below the pixel threshold |

Store `is_color` + `color_score` in the `pages` table. No ML, no downloads,
~1 ms/page on the thumbnail.

### 2. Behavior changes once detected

- Color pages are **auto-deselected** from colorize targets (they are already
  done); UI shows a 🎨 badge on them; export uses the original.
- A banner appears: "N color pages detected — using them as color reference"
  with an off toggle.
- CLI: `--auto-ref on|off` (default on when color pages exist).

### 3. Reference use, per mode (three tiers)

| Tier | Mode | Mechanism | Effort |
| --- | --- | --- | --- |
| 0 — ship first | `fast`, `theme` | Palette anchor: `match_palette` (already exists) pulls every output page toward the LAB moments of the **nearest** color page instead of a user-picked anchor page | small |
| 0 — ship first | `ai` | IP-Adapter auto-ref: instead of requiring `--ref`, feed the detected color page as `ip_adapter_image` (existing code path — only the source of the image changes) | small |
| 1 | `ai` | Nearest-ref selection: embed all color pages once with the CLIP image encoder that IP-Adapter already loads; per B&W page (or per panel) pick the most similar color page as its ref — characters get matched to the color page they actually appear on | medium |
| 2 — Milestone 4 | Cobra reference mode | The detected color pages ARE the reference library Cobra was chosen for (200+ refs supported, 9.3 GB @ 24 refs). Retrieval index from Tier 1 carries over unchanged | the Cobra integration itself |

Ordering matters: Tier 0/1 are pure wiring around existing components
(match_palette, IP-Adapter, CLIP encoder) and work today on 8 GB; Tier 2 is
the quality ceiling.

### 4. Implementation steps (Tier 0 + 1)

1. `pipeline/colorstats.py`: `chroma_stats(rgb) -> (colorful_frac, hue_spread)`, `is_color_page(...)`. Unit tests: synthetic color page, sepia-tinted page, B&W page with JPEG noise, screentone page.
2. Ingest/jobs: compute during thumbnail pass; add `is_color INTEGER, color_score REAL` to `pages` (additive schema migration).
3. jobs.run: build `refs = [color pages]`; when `auto_ref` and refs exist — `fast`: per-page nearest ref (LAB-moment distance) as `anchor_rgb`; `ai`: nearest ref as `ref_rgb` in `ColorizeRequest.extra`; skip color pages from targets unless explicitly selected.
4. Server/UI: `is_color` in pages payload → 🎨 badge + auto-deselect + banner toggle; `auto_ref` in RunJob.
5. CLI: `--auto-ref`, doctor note.
6. Tier 1: cache CLIP embeddings of refs in the job dir; cosine pick per page/panel.

### 5. Edge cases

- Only the cover is color and it is a stylized painting → palette anchor at low strength (0.3) so it nudges rather than repaints; IP-Adapter scale stays at the existing 0.65.
- Color pages of a DIFFERENT character set (pin-up inserts) → Tier 1 nearest-ref naturally down-weights them; worst case the user toggles auto-ref off.
- Spread-split halves inherit the parent page's `is_color`.
- A book that is entirely color → nothing to colorize; UI says so instead of running.

## Final Recommendation

Build InkedIn as an offline-first Tauri desktop app with a Rust trust boundary and a supervised Python ML worker. Use PyTorch CUDA/MPS/ROCm where it is the best-supported path, use ONNX Runtime for DirectML/CoreML/OpenVINO where model export is reliable, and always provide CPU fallback. Treat file ingestion and model loading as the highest-risk areas, isolate them, test them aggressively, and avoid open ports, broad filesystem permissions, telemetry, and arbitrary model code.
